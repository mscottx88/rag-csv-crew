## Requirement Completeness

- [x] CHK001 - Are requirements defined for what happens when the LLM produces a malformed strategy block (e.g., missing delimiters, invalid strategy name, empty SQL)? [Gap, Spec §FR-016] — **Resolved**: Added FR-020 (malformed output handling: skip invalid blocks, proceed with valid ones) and updated contracts/multi-strategy-sql-generation.md with parsing rules.
- [x] CHK002 - Is the behavior specified when the LLM produces fewer strategy blocks than requested (e.g., asked for 3, produces 2)? [Gap, Spec §FR-016] — **Resolved**: FR-020 specifies "proceed with whatever valid blocks were extracted." FR-016 adds retry + fallback for zero valid blocks.
- [x] CHK003 - Are requirements defined for how ctid is stripped from the final user-visible result (users should not see ctid in HTML output)? [Gap, Spec §FR-008/FR-013] — **Resolved**: FR-013 updated: "ctid column MUST be excluded from user-visible output." contracts/response-generator.md specifies ctid exclusion happens in ResultFusionService before response generation.
- [x] CHK004 - Is the behavior specified when strategies return different column sets (e.g., structured returns col1,col2 but FTS returns col1,col3,rank)? [Gap, Spec §FR-007] — **Resolved**: FR-007 updated: "union of user data columns, excluding system columns; missing values set to NULL."
- [x] CHK005 - Are requirements defined for how system-generated columns (_ts_, _emb_, rank, similarity) are handled in fused results? [Gap, Spec §FR-007] — **Resolved**: FR-007 updated: explicit exclusion list "ctid, columns prefixed with _ts_, _emb_, and computed columns like rank, similarity, relevance."
- [x] CHK006 - Is the maximum total fused row count specified after deduplication (currently only per-strategy LIMIT 50 is defined)? [Gap, Spec §FR-011] — **Resolved**: FR-011 updated: "maximum total fused row count is bounded by 3 × 50 = 150 rows. No additional post-fusion limit applied."
- [x] CHK007 - Are requirements defined for how the multi-strategy SQL is stored in the existing `generated_sql` field of the Query model? [Gap, Spec §FR-016] — **Resolved**: Added FR-022: "store multi-strategy SQL output (all blocks concatenated with delimiters) in generated_sql field."
- [x] CHK008 - Is the behavior specified for the case where ctid column is missing from a strategy's SELECT results? [Gap, Spec §FR-008] — **Resolved**: FR-008 updated: "If a strategy's result is missing the ctid column, that strategy's results are excluded from fusion and treated as failed (graceful degradation per FR-012)."
- [x] CHK009 - Are progress/status update requirements defined for multi-strategy execution (user sees "processing" — should they see per-strategy status)? [Gap] — **Resolved**: Added FR-023: progress messages include "Generating multi-strategy SQL...", "Executing N strategies...", "Fusing results...", "Generating response...". Per-strategy completion not reported individually.
- [x] CHK010 - Is the behavior specified when aggregation detection yields a false positive (user asks "how many categories of wireless chargers" — intent is listing, not counting)? [Gap, Spec §FR-019] — **Resolved**: FR-019 updated with explicit note: "false positives acceptable because structured strategy still produces a valid result — user may rephrase." New edge case added.

## Requirement Clarity

- [x] CHK011 - Is "applicable query strategies" in FR-001 precisely defined with explicit inclusion/exclusion criteria for each strategy type? [Clarity, Spec §FR-001] — **Resolved**: FR-001 updated with exhaustive list: "structured (always applicable with B-tree), fulltext (applicable when GIN/FTS indexes exist), vector (applicable when HNSW/vector indexes exist). These three are the exhaustive set."
- [x] CHK012 - Is "relevant columns" in FR-003 and FR-004 precisely defined — does it mean columns in hybrid search results, or all columns with matching indexes? [Clarity, Spec §FR-003/FR-004] — **Resolved**: FR-003 and FR-004 updated: "columns identified by the hybrid search phase (column discovery results)."
- [x] CHK013 - Is the RRF rank assignment clarified — are ranks 1-indexed per the formula, and what rank is assigned when a row is absent from a strategy? [Clarity, Spec §FR-009] — **Resolved**: FR-009 updated: "rank_in_strategy is the 1-indexed position. Rows absent from a strategy receive no score contribution (not included in the sum)."
- [x] CHK014 - Is "per-strategy timeout" in NFR-002 unambiguous — does it mean wall-clock time from SQL dispatch to result, or does it include SQL generation time? [Clarity, Spec §NFR-002] — **Resolved**: NFR-002 updated: "wall-clock time from SQL dispatch to result retrieval (excluding SQL generation time, which is shared)."
- [x] CHK015 - Is "brief strategy attribution summary" in FR-014 specified with enough detail for consistent implementation (format, placement, wording)? [Clarity, Spec §FR-014] — **Resolved**: FR-014 updated with format: "single line above data table: 'Results from {strategy1} ({N1} rows)...' with human-readable names." contracts/response-generator.md created with full specification.
- [x] CHK016 - Is "detect aggregation intent" in FR-019 specified with an explicit list of detection heuristics (keywords, patterns) or is it left to implementer judgment? [Clarity, Spec §FR-019] — **Resolved**: FR-019 updated with keyword list: "count, sum, total, average, avg, minimum, min, maximum, max, how many, what is the total/average/sum (case-insensitive)."
- [x] CHK017 - Is "labeled SQL blocks" in FR-016 specified with a concrete delimiter format for reliable parsing? [Clarity, Spec §FR-016] — **Resolved**: FR-016 updated: delimiters are `---STRATEGY: {name}---` / `---END STRATEGY---`. Regex pattern defined in contracts/multi-strategy-sql-generation.md.
- [x] CHK018 - Is "single LLM call" in FR-016 unambiguous — does it preclude retry logic if the LLM output is unparseable? [Clarity, Spec §FR-016] — **Resolved**: FR-016 updated: "If output is completely unparseable, retry once. If retry fails, fall back to single-strategy structured SQL." Retry logic added to contracts/multi-strategy-sql-generation.md.

## Acceptance Criteria Quality

- [x] CHK024 - Is SC-001 ("improving recall") measurable — what baseline and metric quantify the recall improvement? [Measurability, Spec §SC-001] — **Resolved**: SC-001 updated with measurement method: "compare distinct row counts: multi-strategy should return >= rows of any single strategy alone."
- [x] CHK025 - Is SC-002 ("within 30% of single-strategy baseline") testable — is the baseline measurement method specified (same query, same data, single strategy only)? [Measurability, Spec §SC-002] — **Resolved**: SC-002 updated: "measured on same dataset with same query text. Baseline: single structured SQL. Test: multi-strategy dispatch + parallel execution + fusion."
- [x] CHK026 - Is SC-005 ("no duplicate rows") verifiable — is the definition of "duplicate" clearly tied to ctid matching only, or could different column projections per strategy create apparent non-duplicates? [Measurability, Spec §SC-005] — **Resolved**: SC-005 updated: "no rows with same ctid value. Verified by len(set(ctids)) == len(ctids)."
- [x] CHK027 - Are US1's acceptance scenarios testable without implementation knowledge — does scenario 1 specify how to verify "parallel" dispatch vs. sequential? [Measurability, Spec §US1 AS1] — **Resolved**: US1 AS1 updated: "verified by observability logs showing all three strategies dispatched with overlapping execution windows."
- [x] CHK028 - Is US2 AS1's expected ordering {B, C, A, D} verifiable — does it match the RRF formula with k=60 for 2-strategy results? [Measurability, Spec §US2 AS1] — **Resolved**: Yes, it matches. B at ranks (2,1): 1/62+1/61=0.03252. C at ranks (3,2): 1/63+1/62=0.03200. A at rank (1,absent): 1/61=0.01639. D at rank (absent,3): 1/63=0.01587. Order: B>C>A>D. Verified correct.
- [x] CHK029 - Are US3's acceptance scenarios specific enough — does AS1 "brief note indicating which strategies contributed" define format or content? [Measurability, Spec §US3 AS1] — **Resolved**: US3 AS1 updated: "attribution note above the table listing each contributing strategy and its row count (e.g., 'Results from structured query (12 rows) and full-text search (18 rows)')."

## Scenario Coverage

- [x] CHK030 - Are requirements defined for how the system behaves when the user has no datasets at all (not just empty results, but no datasets)? [Coverage, Gap] — **Resolved**: New edge case added: "The system returns the existing 'no datasets found' response before reaching the multi-strategy pipeline."
- [x] CHK031 - Are requirements defined for the transition path — does the system gracefully handle queries that were in-flight when this feature is deployed? - [x] CHK032 - Are requirements defined for cross-dataset queries where tables have different column schemas (e.g., FTS on table A's "description" but no "description" on table B)? [Coverage, Spec §US4 AS3] — **Resolved**: Edge case updated: "each strategy's SQL targets only the tables/columns where its indexes exist."
- [x] CHK033 - Are requirements defined for what happens when the connection pool is exhausted during parallel strategy execution (3 concurrent connections needed)? [Coverage, Gap] — **Resolved**: New edge case added: "pool.connection() blocks until available. If pool timeout expires, affected strategy fails → graceful degradation." New assumption: "pool max_size >= 5."
- [x] CHK034 - Is the single-strategy fallback path explicitly defined — when only structured strategy applies, does the system bypass fusion entirely or still produce a FusedResult? [Coverage, Spec §FR-002] — **Resolved**: Added FR-021: "bypass multi-strategy prompt and fusion pipeline entirely, using existing single-strategy flow."
- [x] CHK035 - Are requirements defined for cancellation during multi-strategy execution — if the user cancels, are all strategy threads stopped?
