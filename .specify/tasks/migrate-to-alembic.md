# Task: Migrate to Alembic for Database Migrations

## Overview

Replace custom migration scripts with Alembic, the industry-standard database migration tool for Python/SQLAlchemy projects. This provides version-controlled, reversible migrations with proper dependency tracking.

## Current State

**Problems with Current Approach:**
- Custom migration script (`backend/scripts/add_progress_message_column.py`) is ad-hoc
- No version control for migrations
- No rollback capability
- No migration dependency tracking
- Schema definitions split between `migrations.py` and manual scripts
- No way to track which migrations have been applied
- Difficult to coordinate migrations across dev/staging/prod environments

**Current Migration Infrastructure:**
- `backend/src/db/migrations.py` - Manual schema initialization
- `backend/src/db/schemas.py` - SQL template strings
- `backend/scripts/add_progress_message_column.py` - One-off migration script

## Goal

Implement Alembic for all database migrations with:
- Version-controlled migration files
- Automatic schema change detection (autogenerate)
- Up/down migration support (forward and rollback)
- Migration history tracking in database
- Environment-specific configurations (dev, test, prod)

## Implementation Steps

### Phase 1: Alembic Setup

**1.1 Install Alembic**
```bash
cd backend
pip install alembic
pip freeze > requirements.txt
```

**1.2 Initialize Alembic**
```bash
cd backend
alembic init alembic
```

This creates:
- `alembic/` directory with:
  - `env.py` - Migration environment configuration
  - `versions/` - Migration version files
  - `script.py.mako` - Migration template
- `alembic.ini` - Alembic configuration file

**1.3 Configure Alembic**

Edit `alembic.ini`:
```ini
# Use DATABASE_URL from environment
sqlalchemy.url = driver://user:pass@localhost/dbname

# Point to logging configuration
[loggers]
keys = root,sqlalchemy,alembic

[handlers]
keys = console

[formatters]
keys = generic
```

Edit `alembic/env.py`:
```python
import os
from logging.config import fileConfig

from sqlalchemy import engine_from_config, pool
from alembic import context

from dotenv import load_dotenv

# Load environment variables
load_dotenv()

# Import your models' metadata
from src.models import Base  # Or wherever your models are defined

target_metadata = Base.metadata

# Override sqlalchemy.url from environment
config = context.config
config.set_main_option('sqlalchemy.url', os.getenv('DATABASE_URL'))

# ... rest of env.py
```

### Phase 2: Convert Existing Schema to Alembic

**2.1 Create Initial Migration**

Generate baseline migration from current database state:
```bash
alembic revision --autogenerate -m "initial schema"
```

This creates `alembic/versions/xxxx_initial_schema.py`

**2.2 Review and Edit Initial Migration**

The autogenerated migration may not capture:
- User schemas (michael_schema, etc.) - Add manually
- pgvector extension - Add `op.execute("CREATE EXTENSION IF NOT EXISTS vector")`
- Custom CHECK constraints
- Indexes and their specific types (HNSW, GIN)

Edit the migration file to include:
```python
def upgrade():
    # Enable pgvector extension
    op.execute("CREATE EXTENSION IF NOT EXISTS vector")

    # Create public.users table
    op.create_table(
        'users',
        sa.Column('username', sa.VARCHAR(50), primary_key=True),
        sa.Column('schema_name', sa.VARCHAR(63), nullable=False, unique=True),
        # ... rest of columns
        schema='public'
    )

    # Create user schemas dynamically?
    # Note: User schemas are created on-demand, not in migrations
    # Only public schema tables should be in migrations

def downgrade():
    op.drop_table('users', schema='public')
    op.execute("DROP EXTENSION IF EXISTS vector")
```

**2.3 Mark Initial Migration as Applied**

If database already has these tables:
```bash
alembic stamp head
```

This marks all migrations up to `head` as applied without running them.

### Phase 3: Create Migration for progress_message

**3.1 Create New Migration**
```bash
alembic revision -m "add progress_message to queries"
```

**3.2 Implement Migration**

Edit `alembic/versions/xxxx_add_progress_message_to_queries.py`:
```python
from alembic import op
import sqlalchemy as sa
from sqlalchemy import text

def upgrade():
    # Get all user schemas
    conn = op.get_bind()
    result = conn.execute(text("""
        SELECT schema_name
        FROM public.users
        WHERE is_active = TRUE
    """))

    for row in result:
        schema_name = row[0]
        # Add progress_message column to each user's queries table
        op.add_column(
            'queries',
            sa.Column('progress_message', sa.TEXT(), nullable=True),
            schema=schema_name
        )

def downgrade():
    # Get all user schemas
    conn = op.get_bind()
    result = conn.execute(text("""
        SELECT schema_name
        FROM public.users
        WHERE is_active = TRUE
    """))

    for row in result:
        schema_name = row[0]
        # Remove progress_message column
        op.drop_column('queries', 'progress_message', schema=schema_name)
```

**3.3 Apply Migration**
```bash
alembic upgrade head
```

**3.4 Test Rollback**
```bash
alembic downgrade -1  # Go back one version
alembic upgrade head   # Apply again
```

```python
def initialize_database(conn: Connection) -> None:
    """Initialize database using Alembic migrations.

    This runs alembic upgrade head programmatically.
    """
    from alembic.config import Config
    from alembic import command

    alembic_cfg = Config("alembic.ini")
    command.upgrade(alembic_cfg, "head")
```

## Benefits of Alembic

**Version Control:**
- All schema changes tracked in Git
- Migration history is auditable
- Can compare migration states across environments

**Safety:**
- Rollback support for failed deployments
- Review migrations before applying
- Autogenerate detects schema drift

**Team Collaboration:**
- Standard tool everyone understands
- Avoid merge conflicts with sequential version numbers
- Clear migration dependencies

**Production Ready:**
- Used by major Python projects (OpenStack, Reddit, etc.)
- Battle-tested in production environments
- Integrates with SQLAlchemy ORM

## Migration Strategy for Per-User Schemas

**Challenge:** User schemas are created dynamically, not during deployment.

**Solution:**
1. **Public Schema Migrations:** Use Alembic for `public` schema only
2. **User Schema Template:** Keep schema templates in code
3. **On-User-Creation:** Apply schema template when user is created
4. **Backfill Migrations:** For existing users, iterate through all schemas

**Example Migration for User Schemas:**
```python
def upgrade():
    # Get all active user schemas
    conn = op.get_bind()
    result = conn.execute(text("""
        SELECT schema_name FROM public.users WHERE is_active = TRUE
    """))

    # Apply change to each user schema
    for row in result:
        schema = row[0]
        # Apply your schema change
        op.add_column('table', Column(...), schema=schema)
```
